# LLM Inference Ablation Studies

This repository contains an inference engineering study for serving the `Qwen/Qwen3-VL-4B-Instruct` multimodal model with vLLM on Modal-hosted GPUs. The work focuses on controlled ablation experiments for an InferTutor-style workload containing short text prompts, long-context prompts, image prompts, and mixed multimodal traffic.

The goal of the project was not only to find a fast configuration, but to explain why that configuration worked and why several plausible alternatives did not. The final report includes the full experiment log, result summary, architecture diagrams, and discussion of failed or rejected configurations.

## Final Result

The best observed mixed-workload configuration was a 4-replica H100 deployment using:

```text
replicas = 4
max_seqs = 32
max_batch_tokens = 2048
prefix_cache = enabled
chunked_prefill = enabled
mode = mixed
users = 120
max_tokens = 96
```

Final best score:

```text
7,084,876
```

Key metrics for the final best run:

| Metric | Value |
|---|---:|
| TTFT p95 | 638.0 ms |
| ITL p95 | 21.4 ms |
| Throughput | 3231.5 chunks/s |
| Error rate | 0.0% |
| Total GPUs | 4 H100 replicas |

## Main Findings

1. The strongest mixed-workload result came from scheduler balance rather than from enabling every advanced feature.
2. `max-batch-tokens=2048` was the best observed batch-token budget for the mixed workload. Larger and smaller values both regressed.
3. Prefix caching and chunked prefill were both important for the final best configuration.
4. The stable concurrency knee for the final 4-replica mixed configuration was around 120 users. Higher user counts produced queueing collapse or errors.
5. N-gram speculative decoding, online FP8 weight quantization, Modal ingress gating, request-class routing, compiled mode, and disaggregated prefill/decode were all tested as additional directions.
6. Request-class routing improved when shifting from a 2+2 text/heavy split to a 1+3 split, but it still did not beat the single tuned 4-replica mixed pool.
7. The disaggregated prefill/decode prototype was documented as an attempted architecture, but it was not pursued further because the available LMCache integration was incompatible with the installed vLLM version.

## Scoring Formula

The experiment harness uses a throughput-normalized latency efficiency score:

```text
score = goodput_chunks_per_second * users / (ttft_p95_seconds * itl_p95_seconds * total_gpus)
```

where:

```text
goodput_chunks_per_second = aggregate_stream_chunks_per_s * (1 - error_rate)
ttft_p95_seconds = ttft_p95_ms / 1000
itl_p95_seconds = itl_p95_ms / 1000
```

This score rewards useful streamed output, sustained concurrency, and low p95 latency, while penalizing failed requests and inefficient GPU usage. It is best interpreted as a comparative engineering metric for this workload rather than as a universal product metric.

## Setup

Create and activate a Python environment, then install local dependencies:

```bash
cd starter_code
pip install -r requirements.txt
```

Authenticate Modal:

```bash
modal token new
```

Create the Hugging Face secret expected by the Modal app:

```bash
modal secret create huggingface HF_TOKEN=<YOUR_HF_TOKEN>
```

The study used Modal-hosted H100 GPUs. Running these experiments can incur GPU costs, so check Modal usage limits before launching large runs.

## Reproducing the Best Run

From `starter_code/`, run:

```bash
python run_infertutor_experiment.py --label mixed-r4-b2048-u120 --gpu-type H100 --replicas 4 --max-seqs 32 --max-batch-tokens 2048 --mode mixed --users 120 --duration 90 --ramp-up 30 --max-tokens 96
```

After a run completes, stop deployed Modal apps that are no longer needed to avoid unnecessary cost.

## Experiment Families

The study covered the following experiment families:

| Family | Purpose |
|---|---|
| Baseline text and mixed runs | Establish working Modal/vLLM deployment and starting scores |
| Batch-token ablations | Test scheduler behavior under different prefill budgets |
| Prefix-cache and chunked-prefill ablations | Measure whether core vLLM optimizations were helping |
| User-density tests | Find the load point where p95 latency collapses |
| Sequence-concurrency tests | Compare active sequence windows such as `seq16`, `seq32`, and `seq48` |
| N-gram speculative decoding | Test whether speculative execution improved short-output mixed serving |
| Online FP8 quantization | Test whether online weight quantization improved efficiency |
| Modal concurrency gate | Test whether ingress limiting improved queue behavior |
| Request-class routing | Split short text requests from long/image requests using separate endpoints |
| Compiled mode | Test whether text-speed optimizations transferred to mixed multimodal traffic |
| Disaggregated serving prototype | Attempt prefill/decode separation and document the integration blocker |

## Notes on Reproducibility

The exact results depend on GPU availability, Modal scheduling behavior, model download state, vLLM version, and transient cloud conditions. The most reliable way to interpret the study is through relative controlled comparisons: each experiment changed one major serving variable while keeping the workload and scoring harness consistent.
